Coding style

block comments

//====... (76 chars total, up to but not pos 77) ...====
// block comment
//======================================================
block format

Indentation 3 spaces, brackets on separate lines

if(cond)
{
   true;
}
else
{
   false;
}

class names 

class ThisIsAClass
{
   public:
      // constructors/destructors
      // public members

   protected:
      // protected members
   private:
      // private members

   protected:
      // protected fields
   private:
      // private fields
};
Each class should have its own .h and .cpp file.
method and function names

this_is_a_function_or_a_method();
Functions outside class definitions and static functions and variables within classes are forbidden, unless REALLY needed. Discuss with others if in doubt.
global function names

modulenameFunctionName(); // e.g. mixerMixBuffer()
Global functions are only needed for assembler linkage. Use of global functions between C-modules should be prohibited. Existance of such functions may be result of bad design.
variables and fields names

UINT my_own_variable; // underscore is optional
UINT *intptr;
Variables of type bool have 'is' prefix attached to them, e.g.
BOOL isvalid;
BOOL is_supported;
This is recommended that variables of pointer types, especially the ones used for assembler interfacing, should have 'ptr' postfix attached to them, underscore is also optional.
UINT *intptr;
Timer *timer_ptr;
Variables used for enumeration are prefixed with 'num':
int num_tracks;
UINT numbytes;
Used variable types (in engine/tracker, not in storage) are:

UINT
- 32 bit unsigned integer.

BOOL
- boolean type represented via 32 bit unsigned. Its acceptable values are TRUE and FALSE. One can rely on that FALSE is always zero and TRUE is always non-zero.

CHAR
- type used for strings, may be defined as 1 byte or 2 byte char. This allows possible use of UNICODE for localizing the application.
development tools considerations

C++ compiler tool is DJGPP/GCC
Assembler tool is NASM

Note: when you want to add a tool to this list, make sure it can produce portable sources/documents/etc.
common design considerations

The application should use native GUI objects wherever it is ported to. Use of specific interfaces is possible but not recommended.
